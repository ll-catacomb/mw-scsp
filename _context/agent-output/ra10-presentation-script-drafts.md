# RA-10: Five-minute presentation script (Problem-First, with cold-open montage)

Total slot: 5 minutes. Target: 3:30 spoken, 1:30 for Q&A.

The opening is a four-slide visual montage: the same prompt fed to four
different LLMs in sequence, each producing its own list of five Red opening
moves. The audience sees the convergence with their own eyes before any
framing from you. Then a fifth slide cuts to the system's surviving menu
on the same scenario — and not a single move from any of the four LLM
lists appears on it.

Taiwan only. Israel scenario stays in the repo for later; not on the deck.

## Materials prep

### Slide 1–4 (cold-open montage): the prompt and the four model responses

Use **the same prompt verbatim across all four models**. Visual identity of
the slides matters here — same font, same layout, only the model badge and
the response change. The single prompt to use:

> ```
> You are a senior People's Liberation Army (PLA) planner.
>
> Scenario: Taiwan Strait crisis, Spring 2028. Tensions have escalated
> over 18 months. Taiwan is in its electoral cycle. US INDOPACOM is
> partially reconstituted post-Ukraine. Allied positioning is
> forward-leaning but not unified.
>
> Propose Red's opening move for the FIRST 7 DAYS of the crisis.
> Give FIVE distinct candidate moves. For each, name:
>   - the action (be specific)
>   - the units involved
>   - the day-by-day timeline (within the 7-day window)
>   - the intended effect
>
> Just the five moves. No preamble.
> ```

Models to query (pick four — visual variety helps):

|---|---|---|
| Claude Opus 4.7 (or 4.6) | claude.ai or API | Anthropic flagship |
| GPT-5 | chatgpt.com or API | OpenAI flagship |
| Gemini 2.5 Pro | gemini.google.com | Google flagship |
| Grok 4 | grok.com | xAI flagship |
| Llama 3.3 70B (alt) | groq.com or together.ai | Open-weight option |
| DeepSeek V3 (alt) | chat.deepseek.com | Open-weight, geographically interesting |

**Recommended**: Claude, GPT, Gemini, Grok. Four flagships, all closed
weights, four distinct training pipelines. The convergence-despite-difference
is the point.

If you want to drive it through the wrapper for a single audit-trail row
per model, the local script is:

```bash
export PATH="$HOME/.local/bin:$PATH"
uv run python -c "
import asyncio
from src.llm.wrapper import logged_completion

PROMPT = '''[paste the prompt above]'''

async def main():
    for model in ['claude-opus-4-7', 'gpt-5', 'gemini-2.5-pro', 'grok-4-latest']:
        r = await logged_completion(
            run_id='demo-modal-baseline',
            stage='demo_modal',
            agent_id='convergence_demo',
            model=model,
            system='',
            user=PROMPT,
            temperature=1.0,
            max_tokens=4096,
        )
        print(f'\\n===== {model} =====\\n{r[\"raw_text\"]}')

asyncio.run(main())
"
```

This produces four responses you can paste into slides. The audit-log row
gives you a fallback "yes, here's the call I made" if anyone in the
audience asks.

**Slide design notes:**
- Each model gets one slide. Same template. Top: model name + logo if
  the licensing is comfortable, else just the name. Body: five
  numbered moves, monospace or near-monospace.
- Trim aggressively. If a model gives three sentences per move, keep
  the action verb + named unit + timeline phrase. The audience
  doesn't read; they pattern-match.
- Color-code or highlight (subtle, after-the-fact, in the *fifth* slide
  reveal) the recurring moves: Pratas seizure, declared exclusion zone,
  cyber on substrate, amphibious coup-de-main, DF-15 strike. Five
  recurring shapes will be visible across four models — that's the
  modal cluster the audience needs to feel.

## Q&A — anticipated questions and tight answers

These are calibrated to the audience. Each answer is two-to-four
sentences; you don't need to remember them verbatim, but the *first
clause* is the move that lands. Practice the first clauses out loud.

### Core architecture

**Q1. Does this predict adversary behavior?**
> No. The system generates candidate moves the human team has not
> considered, drawn from structured exploration of the space the modal
> LLMs do not cover. Whether any specific move would happen is outside
> the system's claim. The wargame designer decides which to stage.

**Q2. How do you validate it?**
> Not in the OR sense — there is no ground truth for "moves the PLA
> would have made in Spring 2028." What is auditable: every prompt,
> every model output, every judgment is on disk; the menu is
> reproducible from the artifacts. Validation, if anyone wants to do
> it, belongs to the wargame designer who decides whether each
> surviving move was worth playing through.

**Q3. What about hallucination?**
> Two-part. The Red personas read from a controlled doctrine corpus —
> markdown plus YAML, version-controlled — so the modal-baseline
> narration isn't pure model imagination. The off-distribution
> generator deliberately leaves doctrine retrieval out, because its
> job is to escape the gap, not stay in it. Hallucination, in a
> narrowly-scoped channel, is the feature. The structured judges
> then test for adjacency to historical precedent and named leverage
> before scoring plausibility.

**Q4. How is this different from asking GPT-5 the question?**
> Three differences in the audit log. One: a single LLM converges;
> the four-model side-by-side at the start was the demonstration.
> Two: the personas are not promptable voices — each has its own
> agent_id, persistent memory across runs, identity seed, and named
> blind spots, in the Park-et-al generative-agent shape. Three: the
> survival filter is structurally adversarial — moves only survive
> if median plausibility is ≥ 4 *and* zero of five judges would have
> generated the move on their own. A single-LLM ask cannot run that
> filter against itself.

**Q5. How does this differ from "can LLMs replicate an adversary"?**
> We don't try to replicate. Replication is the wrong target — if
> the LLM successfully replicates the modal PLA planner, the player
> team learns nothing they didn't already cover in PME. We surface
> what the ensemble *missed*. The would-have-generated check is the
> hinge: a move only enters the menu if no model in the ensemble
> would have generated it from its default set.

### Sharper follow-ups (the ones that test whether you mean it)

**Q6. How do you know the judges aren't just rubber-stamping each other?**
> The judges are mixed across providers — three Anthropic, two
> OpenAI in the current configuration — and each runs independently
> on the same proposal with no exposure to the others' outputs.
> Their structured ratings disagree often; we have judge-by-judge
> calibration in the audit log. The strict filter requires *zero*
> of five judges to have generated the move from their default.
> Rubber-stamping would show up as flat plausibility distributions
> and consistent "I-would-have-generated-this" votes. We test for
> both.

**Q7. If your Red personas are PME-trained too — same training data — won't they converge?**
> They would, if we leaned on persona-as-prompt. We don't. Each
> persona is its own generative agent in the Park-et-al sense:
> identity seed, persistent memory across runs, named blind spots.
> The blind-spots field is load-bearing — it forces the persona's
> proposals to *route around* its own modal default rather than
> through it. Cross-run, the persona memory lets each agent know
> what it has already proposed and what survived; convergence to
> its own past would be visible in the audit log. That's the point
> of running the ensemble persistently rather than ephemerally.

**Q8. Could this be used to validate a service concept?**
> No, and I'd push back on a request to use it that way. The system
> is structurally bad at confirmation — it's tuned to surface
> uncomfortable hypotheses, not to ratify a preferred concept. If a
> service concept depends on Red doing X, this menu is more likely
> to show up Y, Z, and a third thing the concept didn't account for.
> That's a feature; it's also a deliberate misalignment with the
> "validate-the-concept" use of wargames that the literature has
> been pushing back against for fifteen years.

**Q9. What's the failure mode you've seen most?**
> Plausibility flatline. Early runs had judges defaulting to score
> three across the board because the rationale field was free-text
> and the judges weren't forced to commit. The fix was making
> plausibility a *consequence* of structured fields rather than a
> primary score: adjacency-found yes-or-no with a named historical
> precedent, leverage-named yes-or-no with a named instrument, then
> the score. The rate of meaningful score variance roughly tripled.

**Q10. What's the smallest possible deployment of this?**
> One scenario YAML, one doctrine-corpus directory, one cost cap, the
> wrapper, and the audit-log SQLite. The whole stack is open and runs
> on a laptop in ten minutes for about six dollars in model usage. No
> classified material in the demo corpus; everything is open-source
> CSIS / IISS / RAND / IDA. Anyone in this room could fork the repo
> tonight and have a Taiwan run by tomorrow.

**Q11. Why not just use one model with chain-of-thought?**
> Chain-of-thought stays on the modal default — same model, same
> training distribution, same five answers with more words around
> them. The convergence problem isn't a reasoning problem; it's a
> distribution problem. You don't escape a distribution by reasoning
> harder inside it. You escape it by introducing structure that
> forces the system to test whether what it just produced was *also*
> what it would have produced absent the structure. That's what the
> would-have-generated check does.

**Q12. How does this scale to a longer wargame? You only show the first 7 days.**
> The 7-day window is a scenario parameter, not a system limit. Scale
> in two directions. One: each surviving move ships with a
> self-contained context pack — about 30 KB of markdown,
> persona-grounded — that can be re-fed to a continuing model
> instance when the wargame moves into player counter-moves.
> Two: the curator stage is per-branch and the persona memory is
> per-agent, so an exercise running across three days can re-run the
> pipeline against the player team's actual moves and surface what
> the *next* off-distribution Red move would be given the new state.
> Persistent memory across runs is the lever for that.

**Q13. How much human time does this take?**
> Per run: zero, after the scenario YAML is written. Authoring a new
> scenario is the human work — a couple of hours per scenario for the
> doctrine corpus assembly and the YAML, and that's the slow step. A
> wargame designer who has done one scenario before can spin up a
> new one in an afternoon.

### If a question lands you don't know the answer to

Default response: *"I don't have a clean answer to that yet. Want to
follow up after?"* — and mean it. The audience can tell when you're
faking. Wong's register on this is operative honesty over confident
filler. Saying "I don't know" once does not damage the talk; bluffing
once does.
